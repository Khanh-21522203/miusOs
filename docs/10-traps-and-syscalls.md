# xv6-riscv in Rust: Traps and System Calls (Theory)

This chapter explains how control crosses between user and kernel modes.

## 1. Trap categories
- Exception: generated by current instruction stream (`ecall`, page fault, illegal instruction).
- Interrupt: asynchronous hardware event (timer, UART, disk).

## 2. Key RISC-V CSRs
- `stvec`: trap entry address.
- `scause`: why trap happened.
- `sepc`: program counter at trap.
- `stval`: extra fault info (often bad address).
- `sstatus`: status bits controlling return behavior.

## 3. Syscall flow
```text
User process
  -> places syscall number in a7, args in a0-a5
  -> executes ecall
CPU trap hardware
  -> switches to S-mode
  -> jumps to stvec
Trap entry asm
  -> saves registers to TrapFrame
  -> calls Rust trap dispatcher
Rust dispatcher
  -> if ecall: route to syscall table
  -> if timer/uart: handle interrupt
  -> else: fault handling
Trap return asm
  -> restores registers
  -> sret back to user
```

## 4. Syscall ABI contract
Example convention:
- `a7`: syscall number.
- `a0-a5`: up to 6 args.
- Return value in `a0`.

User pointers are untrusted. Kernel must validate every user address range.

## 5. Dispatch strategy
Rust enum for cause decoding:
```rust
pub enum TrapCause {
    UserEcall,
    PageFault { addr: usize },
    TimerInterrupt,
    ExternalInterrupt,
    Unknown(u64),
}
```

## 6. Trapframe ownership
Each running task has a trapframe that stores user register context across trap handling.

Invariants:
- Trapframe is kernel-owned memory.
- User code never writes trapframe directly.
- Return path restores trapframe exactly once.

## 7. Common failure points
- Wrong `stvec` address.
- Clobbered registers in assembly entry/exit.
- Failing to increment `sepc` after syscall.
- Returning to user with interrupts misconfigured.
